# Pensieve — Resumen de exploración (2026-07-15)

## Qué es

Editor de notas que guarda cada nota en **tu repo privado de GitHub**: el
historial de cambios son commits, funciona offline, y usa el mismo editor que VS
Code (Monaco). App madura — el README cuenta 4 años de uso y 91 MB de notas del
autor. Vive en `pensieve.amatiasq.com`.

- **Frontend**: React 19 + Vite 6 + Emotion (CSS-in-JS) + Monaco. PWA
  (`vite-plugin-pwa`), offline vía IndexedDB (`idb-keyval`). ~6.500 líneas.
- **Backend**: un **Cloudflare Worker** (`src/api/`, en JS) con dos rutas:
  `/auth` (intercambio OAuth de GitHub) y `/commit` (proxy para commitear al
  repo).
- **Tests**: 11 specs e2e de Playwright (create-delete, editor, offline, sync…).
- **Docs**: `docs/ARCHITECTURE.md`, `OVERVIEW.md` — buenas y al día.

## Arquitectura (bien pensada)

El código está en capas numeradas `0-dom` … `7-components`, con una regla clara:
cada capa sólo importa de capas de número igual o inferior. De abajo arriba: DOM
→ core (message bus, HTTP) → entities (Note, Settings) → github → storage → app
→ hooks → components. El flujo de datos (Editor → hook → RemoteNote →
MixedStore → local IndexedDB + remoto GitHub) está documentado con diagramas.

## Hallazgos

### Seguridad (el Worker de `src/api/`)

1. **La validación de origen no valida nada.** `handleCorsRequest` saca el origen
   de `new URL(request.url).origin` — que es la URL del **propio worker**
   (`pensieve-api.amatiasq.workers.dev`), no el header `Origin` del navegador que
   llama. Ese valor está hardcodeado en la whitelist, así que `isValidOrigin`
   **siempre pasa**, venga la petición de donde venga. Y encima responde
   `Access-Control-Allow-Origin: '*'`. La lista `VALID_ORIGINS` (de config) no
   protege nada. Impacto acotado (el `/commit` necesita el token del usuario en el
   body, que no es accesible cross-origin), pero la puerta está mal cerrada.

2. **`auth.js` usa secretos que no importa.** Referencia `CLIENT_SECRET_DEV` /
   `CLIENT_SECRET_PROD`, pero la línea 1 sólo importa `CLIENT_ID_*` de
   `config.json`, y `config.json` no contiene ningún `CLIENT_SECRET` (verificado).
   O el intercambio OAuth está roto, o los secretos llegan por un mecanismo no
   visible (wrangler secrets). En un worker de formato módulo los secrets llegan
   por el parámetro `env`, que `auth` no recibe. **Hay que verificar que el login
   funciona y de dónde salen esos valores.**

3. **`console.log` en producción**: el worker loguea el origen en cada request
   (`console.log({ origin })`) y `owner/repo/branch` en cada commit.

4. **El token de GitHub vive en `localStorage`** (`GithubAuth`). Es el modelo
   estándar de una SPA, pero ese token da **escritura** al repo; cualquier XSS lo
   llevaría. Nota, no acción inmediata.

### La red de seguridad está a medias (patrón del mono, otra vez)

5. **El CI sólo corre `typecheck`.** No corre `lint` (que hoy da **10 errores**),
   ni los **11 specs e2e**, ni el `build`. `tsc` pasa limpio, pero el resto de la
   red — que existe y es buena — no se ejecuta, así que se degrada sin que nadie
   lo vea. Es la misma historia que mud, conta y jarvis-api.

### Limpieza

6. **10 errores de lint** acumulados, todos triviales (variables sin usar,
   expresiones sin efecto, `self` no declarado en el service worker por config de
   eslint).

7. **Dos lockfiles** en git (`bun.lock` + `package-lock.json`).

## Estado verificable

- `tsc --noEmit`: **limpio**.
- `eslint`: **10 errores**.
- CI (`ci-pensieve.yml`): **sólo typecheck**.
- e2e: 11 specs, no ejecutados en CI (no los corrí aquí — necesitan servidor +
  auth).
- `.env` **no** está en git (bien); contiene los client secrets de OAuth en local.

## Propósito: claro

No necesito preguntar nada. README y `docs/` explican qué es y cómo está hecho
con detalle. El proyecto está **bien construido**; los problemas son de borde
(el worker) y de proceso (el CI mira sólo una parte), no de diseño.
