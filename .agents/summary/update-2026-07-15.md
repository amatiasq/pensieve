# Pensieve — Informe de mejoras (2026-07-15)

Auditoría a fecha de hoy. Nada tocado: esto es el informe. Dentro de cada
categoría, primero lo que más devuelve por lo que cuesta.

pensieve es un proyecto **bien construido**: arquitectura en capas con regla de
dependencias, docs excelentes, e2e reales, offline/sync sofisticado. Los

## 0. Seguridad — el Worker (`src/api/`)

### 0.1 · Validar el origen de verdad

- **Qué**: `handleCorsRequest` lee el origen de `new URL(request.url).origin`,
  que es la URL del propio worker, no del cliente. Como ese valor está en la
  whitelist, `isValidOrigin` **siempre pasa**, y encima la respuesta lleva
  `Access-Control-Allow-Origin: '*'`. La lista `VALID_ORIGINS` no filtra nada.
- **Dónde**: `src/api/index.js:19-23,32,59-65`
- **Por qué**: el CORS es la única barrera del worker. Hay que leer el header
  `Origin` del cliente, validarlo contra la whitelist, y reflejar **ese** origen
  (no `*`) en `Access-Control-Allow-Origin`. El impacto real hoy es limitado
  (el `/commit` exige el token del usuario en el body), pero una barrera que no
  filtra es peor que no tenerla: da falsa confianza.
- **Esfuerzo**: S · **Riesgo**: bajo — **verificar que el login y el commit
  siguen funcionando desde el origen real tras el cambio**

```js
// index.js — antes: el origen del propio worker, siempre válido
const { origin } = new URL(request.url);
// ...
response.headers.set('Access-Control-Allow-Origin', '*');

// después: el origen del cliente, y sólo si está permitido
const origin = request.headers.get('Origin') ?? '';
const allowed = isValidOrigin(origin);
// ...
if (allowed) response.headers.set('Access-Control-Allow-Origin', origin);
response.headers.set('Vary', 'Origin');
```

### 0.2 · Hacer explícito de dónde salen los client secrets

- **Qué**: `auth.js` usa `CLIENT_SECRET_DEV` / `CLIENT_SECRET_PROD` como
  identificadores globales, pero sólo importa `CLIENT_ID_*` de `config.json`, y
  `config.json` no contiene ningún secret. **El login funciona a diario** (dato
  del autor), así que los secretos llegan por un camino no visible en el código
  — casi seguro `wrangler secret put`, expuestos como globals.
- **Dónde**: `src/api/auth.js:1,15` · `src/api/index.js:8`
- **Por qué**: no es un bug — es magia invisible. El worker es formato módulo, y
  lo idiomático es recibir los secretos por el parámetro `env` del `fetch` y
  leerlos como `env.CLIENT_SECRET_*`, no como globals que aparecen de la nada.
  Cablear `env` por la cadena (`fetch(request, env) → handler(request, env)`)
  hace el código legible y deja de depender de un comportamiento de runtime que
  no está escrito. No urge, pero cierra una duda que cuesta cinco minutos.
- **Esfuerzo**: S · **Riesgo**: bajo

### 0.3 · Quitar los `console.log` del worker

- **Qué**: loguea el origen en cada request y `owner/repo/branch` en cada commit.
- **Dónde**: `src/api/index.js:21,48` · `src/api/commit.js`
- **Por qué**: ruido en los logs de Cloudflare, y registra a qué repos commitea el
  usuario. Quitar o degradar.
- **Esfuerzo**: S · **Riesgo**: bajo

## 1. La red de seguridad — que el CI corra lo que ya existe

### 1.1 · Añadir lint, build y e2e al CI

- **Qué**: `ci-pensieve.yml` sólo corre `typecheck`. El proyecto tiene `lint`
  (hoy con 10 errores), `build`, y 11 specs e2e — nada de eso corre en CI.
- **Dónde**: `.github/workflows/ci-pensieve.yml`
- **Por qué**: la red existe y es buena; sin ejecutarla se degrada en silencio
  (el lint ya lo hizo). Es el mismo patrón que en mud, conta y jarvis-api.
- **Los e2e ya no están bloqueados por la auth.** El autor recordaba no haber
  podido pasar el OAuth de GitHub en los e2e, pero `e2e/fixtures.ts` ya lo
  resuelve: intercepta el endpoint `/auth` del worker
  (`page.route(...'/auth**')`) e inyecta un token falso en `localStorage`
  (`notes.gh-token`). Así que el bloqueo que impedía correrlos ya está superado en
  el código — corren sin OAuth real.
- Orden sugerido: arreglar los 10 errores de lint (1.2), añadir `lint` y `build`
  al job (baratos y rápidos), y los e2e en su propio job (necesitan navegadores
  de Playwright y el server; van bien en cada push, pero separados para no frenar
  el feedback de lint/build).
- **Esfuerzo**: S (lint+build) / M (e2e) · **Riesgo**: bajo

### 1.2 · Arreglar los 10 errores de lint

- **Qué**: variables sin usar (`key`, `error`), expresiones sin efecto, y `self`
  no declarado en el service worker (config de eslint, no bug real).
- **Dónde**: `serialization.ts`, `Stopwatch.ts`, `Editor.tsx`, `Resizer.tsx`,
  los tres middlewares de storage, y el service worker.
- **Por qué**: son triviales y bloquean 1.1. Para `self`, añadir el env de
  service worker / webworker a la config de eslint en vez de silenciarlo caso a
  caso.
- **Esfuerzo**: S · **Riesgo**: bajo

## 1b. La regla de capas (lo que siempre quisiste)

### 1b.1 · Una regla ESLint que impida importar hacia arriba

- **Qué**: las carpetas están numeradas `0-dom` … `7-components` justo para que
  cada capa sólo importe de capas de número igual o inferior — pero hoy **nada lo
  hace cumplir**. Es una convención de honor. El autor siempre quiso la regla que
  la vuelva mecánica.
- **Dónde**: `eslint.config.js`
- **Por qué**: es la regla que convierte la arquitectura documentada en
  arquitectura garantizada. Un import de `2-entities` a `5-app` deja de pasar el
  lint en vez de descubrirse por casualidad meses después.
- **Esfuerzo**: M · **Riesgo**: bajo

  **Cómo, fiel al espíritu artesanal (cero dependencias nuevas):** en la flat
  config de ESLint 9 se puede declarar un plugin local con una regla hecha a mano.
  El número de capa sale del primer segmento de la ruta bajo `src/`; se compara el
  del fichero con el del import y se marca si el import apunta a una capa mayor.

  ```js
  // eslint.config.js — plugin local, ~25 líneas, sin instalar nada
  const layerOf = (p) => {
    const m = p.replace(/.*\/src\//, '').match(/^(\d)-/);
    return m ? Number(m[1]) : null;
  };

  const layers = {
    rules: {
      'no-upward-import': {
        create(context) {
          const self = layerOf(context.filename);
          if (self == null) return {};
          return {
            ImportDeclaration(node) {
              // sólo imports relativos entre capas de src/
              const to = layerOf(
                new URL(node.source.value, `file:///src/${self}-x/`).pathname,
              );
              if (to != null && to > self) {
                context.report({
                  node,
                  message: `Layer ${self} must not import from layer ${to}.`,
                });
              }
            },
          };
        },
      },
    },
  };

  // ...en la config:
  { plugins: { layers }, rules: { 'layers/no-upward-import': 'error' } }
  ```

  El detalle fino es resolver la ruta del import a su capa (los `../` cruzan
  carpetas); la forma robusta es usar el `resolve` del fichero real, o apoyarse en
  `import/no-restricted-paths` de `eslint-plugin-import` si prefieres no mantener
  la resolución a mano. Pero la versión local, sin dependencia, es la que encaja
  con "programado a mano" — y una vez escrita, no se toca.

  **Alternativa con plugin** (si la resolución a mano da guerra):
  `eslint-plugin-boundaries` está hecho exactamente para esto — defines los
  `element-types` por patrón de carpeta y una regla de qué capa puede importar a
  cuál. Es una dependencia de dev, pero declarativa y bien probada. Decisión tuya:
  artesanal sin deps, o declarativo con una dep.

## 3. Nota, sin acción inmediata

### 3.1 · El token de GitHub en `localStorage`

- **Qué**: `GithubAuth` guarda el token en `localStorage`. Da escritura al repo, y
  un XSS lo llevaría.
- **Dónde**: `src/3-github/GithubAuth.ts`
- **Por qué**: es el modelo estándar de una SPA y cambiarlo es grande (requiere
  que el token no toque el JS del cliente — p. ej. cookie httpOnly gestionada por
  el worker, lo que cambia el flujo). No lo propongo como cambio; lo dejo escrito
  como decisión consciente, igual que la contraseña en localStorage de mud.
- **Esfuerzo**: L · **Riesgo**: — · **no lo tocaría sin una razón**
